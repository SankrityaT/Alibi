# Alibi — Design Spec

Date: 2026-07-10
Hackathon: Supermemory Local (localhost:6767), submission deadline 2026-07-13 23:59 PST

## 1. Concept

Alibi is a noir detective game where every suspect has real, isolated, persistent
memory, powered entirely by a local Supermemory instance. The player walks a
police station and interrogates suspects, inspects evidence, plants true or
false information, and makes accusations. No suspect dialogue is scripted or
branched by hand — every answer is generated by Claude, grounded only in what
that suspect's own memory search returns. The product being demonstrated is
the memory layer, not the mystery; the mystery is the vehicle that makes the
memory layer legible on camera.

## 2. Why this proves Supermemory Local

- **Isolation**: each suspect has their own `containerTag`. A suspect can only
  answer using memories scoped to them — they cannot know what another
  suspect said or witnessed unless the player tells them.
- **Live write + live read**: player actions (showing evidence, making a
  statement, planting a lie) are written to Supermemory in real time via
  `/v3/documents`, and are immediately retrievable in the very next
  interrogation turn via `/v4/search`.
- **Mutable belief**: a suspect cannot distinguish a true memory from a false
  one planted by the player. Lies persist, get repeated back, and can be
  revealed later — behavior a stateless chatbot cannot produce.
- **Dynamic profile**: `/v4/profile` is used to track each suspect's evolving
  trust/suspicion of the player, shown as a visible before/after diff after
  key moments (being caught lying, being shown evidence).
- **Cross-case persistence**: the detective's own memory is never wiped
  between cases. A later case can visibly retrieve and reference facts from
  an earlier one, proving memory accumulates rather than resetting per
  session.
- **Visible proof, not claimed proof**: every screen that uses memory shows
  its work (see Section 6, Memory Trace Panel). The goal is that a judge
  watching the demo video can see the actual search query and the actual
  memory it returned, not just take the game's word for it.

## 3. Player role and core loop

The player is the detective. Per suspect, per turn, the available actions
are:

1. **Ask a free-form question.** The question is embedded/queried against
   that suspect's `containerTag` via `/v4/search` (hybrid mode). Retrieved
   memories + the suspect's static character sheet are passed to Claude,
   which generates the in-character answer. If nothing relevant is
   retrieved, the suspect genuinely doesn't know and says so.
2. **Show evidence / confront with a statement.** The evidence or statement
   is written as a new memory document into that suspect's container
   (`/v3/documents`), tagged with a timestamp and source. It is immediately
   searchable on the next question.
3. **Plant a lie.** Same write path as #2, but the content is false. The
   player either types their own or uses "Suggest a lie," which sends
   current case state (contradictions found so far, this suspect's prior
   statements, other suspects' known statements) to Claude to generate 2-3
   plausible false claims. The player picks or edits one, then confirms —
   the write only happens on confirmation.
4. **Check the case board.** A cross-suspect view surfacing detected
   contradictions between statements, each with a citation back to the
   underlying memory (which suspect, which document, when it was said).
5. **Make an accusation.** Ends the case. The ending text is generated from
   the actual memory trail (who was shown what, who was told what, which
   lies were planted and whether they were ever caught), not a fixed
   branch. There is no hard fail state — an incorrect accusation still
   produces a coherent, consequence-bearing ending, and (per Section 5) that
   consequence can resurface in a later case.

## 4. Memory architecture

Two distinct kinds of content go into each suspect, handled differently:

- **Static character sheet** (not searched): voice, personality, core
  motive, relationships, what they're hiding. A few hundred words, fixed at
  game start, injected into every Claude call for that suspect regardless
  of the question. This is what keeps a suspect's voice consistent turn to
  turn.
- **Atomic ground-truth memories** (searched): what the suspect actually
  witnessed during the case's timeline, ingested as ~10-20 separate,
  specific documents per suspect (not one large blob). Precision over
  volume — a haystack of backstory prose would hurt retrieval quality, not
  help characterization.
- **Dynamic memories** (searched, grows during play): every player action
  during that suspect's interrogation — evidence shown, statements made,
  lies planted — written live and retrievable on the next turn.

Cross-case memory is scoped to the detective, not to any suspect: case
outcomes, accusations, and notable moments are written to a
detective-level container that persists across cases and can be retrieved
during a later case's interrogations.

## 5. Case content

### Case 1 — "The 3AM Courier" (fully playable, this is the MVP)

Theo Marsh, a courier, vanishes mid-delivery. Last GPS ping: Old Docks,
22:10. His van is found abandoned at 03:00, package gone.

Suspects (recurring cast, used again in later cases):

- **Mara Okafor** — dispatcher. Secretly rerouted Theo's path at 21:45
  (dispatch log shows the edit). Claims it was "traffic."
- **Ivo Kessler** — rival courier who wanted Theo's higher-paying route.
  Was at the docks that night; denies it.
- **Priya Anand** — the client. Package was declared as "documents"; the
  warehouse handoff photo shows something shaped like electronics.
- **Jonas Marsh** — Theo's brother, owes Ivo money, was supposed to meet
  Theo that night to borrow cash. Claims he was home.

Evidence (~15 items), including: a train receipt on Jonas's phone placing
him at the docks at 22:00 (contradicts his "I was home" statement), Mara's
dispatch-log edit history, Priya's warehouse handoff photo, Ivo's texts
with an unknown buyer about "the shipment," Theo's last voice memo
("someone's been asking about the package all week").

### Case 2 (scoped small — proves cross-case memory, not a full second mystery)

Opens with one of the same four suspects referencing Case 1's outcome
unprompted — retrieved live from the detective's persisted memory during
that suspect's `/v4/search` call. This is a single interrogation scene, not
a full standalone case, and exists specifically to demonstrate that memory
survives and matters across sessions. Building it out into a full second
mystery is a stretch goal only if time remains after Case 1 and the memory
trace/profile UI are solid.

A third case is explicitly out of scope for the hackathon deadline.

## 6. UI and visual design (high priority — this is a stated priority for judging)

The UI carries the entire demo, since the mechanic itself (memory calls)
is invisible unless surfaced. Two screen types:

- **Station map (Phaser or canvas).** Top-down illustrated police station:
  hallway, two interrogation rooms, evidence locker, case board room. The
  player has real, controllable movement (arrow keys/WASD), not
  click-to-teleport. Walking into a room's door trigger transitions into
  that room's scene.
- **Room scenes (React + Tailwind + Framer Motion).** Noir visual style:
  dark palette, grain/texture overlay, typewriter-style dialogue reveal,
  evidence cards with a flip/reveal animation, portrait art per suspect
  (static illustration, ken-burns-style slow zoom during interrogation for
  cinematic feel).
- **Memory Trace Panel** (present during interrogation and on the case
  board): shows, per suspect answer, the exact search query fired, the
  memory chunks retrieved (with source + timestamp), and — after key
  moments — the `/v4/profile` before/after diff (e.g. "suspicion of
  player: low → high"). This panel is the core proof-of-mechanic surface
  and should be visually treated as a first-class feature, not a debug
  overlay.
- **Case board**: a node graph (suspects and evidence as nodes, SVG lines
  drawn between contradicting statements when detected), each line
  clickable to show the citation back to source memories.

UI polish is treated as a primary requirement, not a nice-to-have, because
the demo video is judged largely on what's visible on screen in three
minutes.

## 7. Technical stack

- **Frontend**: Next.js, Tailwind CSS, Framer Motion.
- **Station movement layer**: Phaser (or plain canvas if lighter-weight
  suffices) embedded in the Next.js app; room-entry triggers hand off to
  React scenes.
- **LLM**: Claude, via the Anthropic SDK — chosen over GPT because
  narrative/fiction content (in-character dialogue, lie suggestions) is
  handled more reliably without over-triggering content filters, per
  policy check done during design (see conversation; both providers'
  usage policies permit fictional roleplay content, deception restrictions
  target real-world misuse, not in-game fiction).
- **Memory**: Supermemory Local, `localhost:6767`. Endpoints used:
  `/v3/documents` (write memories/evidence/character sheets), `/v4/search`
  (hybrid mode, per-suspect `containerTag`), `/v4/profile` (suspect mental
  state).
- **Deployment**: local-first for the demo recording; a hosted deploy
  (e.g. Vercel) is a stretch goal if it doesn't compromise the "runs
  entirely on your machine" story the hackathon is built around — the
  local Supermemory dependency means a public hosted deploy would need a
  reachable Supermemory instance, which is a genuine open question, not
  assumed to be solved.

## 8. Error handling

- If `/v4/search` returns no relevant memories for a question, the suspect
  responds in-character as genuinely not knowing — this is a normal case,
  not an error state.
- If a Supermemory write fails (e.g. local server not running), the
  action should surface clearly in the UI rather than silently proceeding
  as if the memory was recorded — a silently-lost write would break the
  core mechanic's integrity.
- If Claude's response doesn't ground in the retrieved memories (drifts
  into inventing new facts), the memory trace panel makes this visible
  and inspectable rather than hidden, which is itself useful for
  debugging during build.

## 9. Testing / demo plan

- Manual playtest of the full Case 1 loop (ask, show, lie, accusation)
  before considering it done — no automated test suite is planned given
  the timeline; this is a demo-driven build.
- Explicit playtest of the poison mechanic: plant a lie, confirm it's
  retrievable and repeated back in a later turn, confirm a suspect who was
  never told it does not know it (isolation check).
- Explicit playtest of cross-case memory: complete Case 1, start Case 2,
  confirm the opening scene's suspect reference is a real retrieval, not
  hardcoded text.
- Record the demo video (max 3 minutes) directly from a live playthrough
  covering: case board intro, a lie planted and caught, the memory trace
  panel visibly firing, the Case 2 cross-case callback, and the final
  accusation.

## 10. Explicit non-goals for this hackathon

- No third case.
- No full free-roam movement physics beyond simple room-to-room walking
  with collision against walls/doors.
- No multiplayer, no save/load beyond what Supermemory itself persists.
- No hard win/lose or scoring system — the ending is narrative, not a
  pass/fail grade.
